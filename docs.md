# Проект документации
## Финтех-проект: «Модуль базового фрод-мониторинга»

### 1. Цель проекта

#### Бизнес-цель
Создать модуль автоматической проверки транзакций, который:
- снижает финансовые потери от мошенничества;
- обеспечивает соответствие регуляторным требованиям;
- уменьшает операционные риски;
- повышает доверие клиентов к платформе.

#### Регуляторная цель
Модуль должен обеспечивать:
- автоматическое применение лимитов операций;
- усиленный контроль транзакций из high-risk юрисдикций;
- трассируемость решений (audit trail);
- возможность объяснения причин блокировки.

#### Техническая цель
Реализовать:
- rule-based контроль (лимиты + частотный анализ);
- механизм снижения лимитов для high-risk стран;
- логирование решений;
- расширяемость до ML-модели.

### 3. Основные требования (User Stories)

#### 3.1 Регуляторные требования

**US-1**  
Как комплаенс-офицер,  
я хочу автоматически снижать лимиты для high-risk стран,  
чтобы выполнить требования регулятора.  

*Реализация в прототипе:*  
```python
effective_limit = amount_limit / 2 if is_high_risk_country else amount_limit
```

#### 3.2 Прозрачность решений

**US-2**  
Как оператор поддержки,  
я хочу видеть причину блокировки транзакции в логах,  
чтобы быстро ответить клиенту.  

*Реализация:*  
```python
status = "BLOCKED: High Risk Country & Amount"
```
Каждая транзакция содержит:
- `effective_limit`
- `txns_in_last_hour`
- статус
- признаки high-risk

#### 3.3 Частотный контроль

**US-3**  
Как риск-менеджер,  
я хочу выявлять аномально частые операции,  
чтобы предотвратить автоматизированные атаки.  

*Реализация:*  
```python
is_high_freq = len(dq) > freq_limit
```

#### 3.4 ML-расширяемость

**US-4**  
Как ML-инженер,  
я хочу использовать поведенческие признаки,  
чтобы повысить точность детекции.  

*Реализовано:*  
- `amount`
- `hour_of_day`
- `day_of_week`
- `txns_in_last_hour`
- `merchant_country` (one-hot)

### 4. Риски и митигация

#### 4.1 Ложные срабатывания (False Positive)
**Риск:**  
Блокировка легитимных клиентов → репутационные потери.  

**Митигация:**  
- Введение промежуточного статуса `REVIEW`  
- Использование ML-score вместо жёстких правил  
- A/B тестирование порогов  

#### 4.2 Пропуск мошенничества (False Negative)
**Риск:**  
Финансовые потери и регуляторные санкции.  

**Митигация:**  
- Комбинация rule-based + ML  
- Повышение чувствительности для high-risk стран  
- Мониторинг ROC-AUC  

#### 4.3 Обход правил (Rule Evasion)
**Риск:**  
Разбиение суммы на несколько мелких транзакций.  

**Митигация:**  
- Частотный анализ  
- Sliding window  
- Поведенческая ML-модель  

#### 4.4 Задержки в processing
**Риск:**  
Рост latency платёжного потока.  

**Митигация:**  
- In-memory хранение окна транзакций  
- Асинхронная ML-оценка  
- Кэширование country-risk  

#### 4.5 Регуляторный риск
**Риск:**  
Несоответствие требованиям надзора.  

**Митигация:**  
- Версионирование правил  
- Хранение audit trail  
- Документирование логики

### 5. Связь с аналитической частью (Сравнительная таблица)

#### 5.1 Связь с п.2 — Управление рисками

В сравнении IT vs FinTech:

| Классический IT       | Финтех                              |
| --------------------- | ----------------------------------- |
| Операционные риски    | Финансовые + регуляторные + репутационные |

Прототип иллюстрирует:
- **Финансовый риск** → контроль суммы
- **Операционный риск** → контроль частоты
- **Регуляторный риск** → снижение лимита для risky стран
- **Репутационный риск** → логируемость и объяснимость

Таким образом, код демонстрирует, что в финтехе риск-логика встроена прямо в ядро бизнес-процесса.

#### 5.2 Связь с п.3 — Требования к безопасности

Из таблицы:

| Классический IT       | Финтех                          |
| --------------------- | ------------------------------- |
| Защита данных         | Защита данных + денег           |

В модуле реализованы элементы security-подхода:
- Контроль high-risk юрисдикций
- Ограничение финансового воздействия
- Трассируемость решений
- Минимизация attack surface через частотный контроль

ML-часть дополнительно:
- снижает зависимость от статических правил
- усложняет обход системы
- повышает адаптивность к новым атакам

```

        ↓

        ↓
Final Decision
        ↓
Logging & Audit Trail
```
